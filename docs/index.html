What it does

The tool handles the technical sequence for you.

You supply one public URL. The project instructions tell the agent exactly how to observe the target, extract the design system, build each section, and deliver a maintainable application — not a fragile export.

Inspect

The agent screenshots the page, reads computed styles, records breakpoints, and maps every interaction before writing a single line of code.

Specify

Each section becomes a concrete written spec — exact values, states, and responsive targets — so builders are implementing instructions, not guessing.

Build and assemble

Components are built from their specs, then wired together into an app-router Vite project ready for you to extend immediately.

Start here

Follow these five steps in order.

If this is your first time, work through each step one at a time. Confirm it worked before moving to the next.

  1. 1

    Create your own copy of the repository

    Open the repository on GitHub and create your own writable copy. You need a copy that belongs to your account so you can commit, push, and track changes as the agent builds.

    • Go to the repository and click Use this template or use the fork flow.
    • Give your copy a name you will recognize — for example, my-clone-project.
    • Wait for GitHub to finish creating your new repository before continuing.
  2. 2

    Clone the repo locally and install dependencies

    Bring your copy to your machine. The AI agent works against the local checkout, not directly on GitHub. 

    git clone https://github.com/YOUR-USERNAME/YOUR-REPO-NAME
cd YOUR-REPO-NAME
pnpm install
    • Replace YOUR-USERNAME with your GitHub username.
    • Replace YOUR-REPO-NAME with whatever you named the repo.
    • If you do not have pnpm yet: npm install -g pnpm
  3. 3

    Launch your AI agent with browser access enabled

    The cloning workflow needs the agent to open the target website in a real browser session so it can capture screenshots and inspect live computed styles. Start your agent from inside the repository directory. 

    # Claude Code (recommended — most tested)
claude --chrome

# Other supported agents: Cursor, Windsurf, Gemini CLI,
# Codex CLI, Continue, Amazon Q, Augment, OpenCode
    • Start the agent from the repository root — not a subfolder.
    • Keep the terminal open while the workflow is running.
    • All supported agents read AGENTS.md automatically.
  4. 4

    Run the clone command

    Type the slash command followed by the full public URL of the site you want to reproduce. That single command starts the entire five-phase pipeline. 

    /clone-website https://example.com
    • Use a complete URL including https://.
    • The site must be publicly accessible — no login required for the agent to open it.
    • The agent will narrate each phase as it runs. You do not need to intervene until it finishes.
  5. 5

    Review the output and start the dev server

    After the workflow completes, launch the app locally to see what was built. Review the generated files before making any changes of your own. 

    pnpm dev
    • Open http://localhost:5173 in your browser.
    • Check src/App.tsx to see how sections are composed.
    • Check src/index.css to see how tokens were translated.
    • Inspect docs/research/ to read the spec files from the cloning run.
Pipeline

What the agent does after you run the command

The agent does not jump straight into writing components. It follows a staged process so every build decision is grounded in evidence collected from the real site.

Phase 1 — Recon

  • Opens the target URL in a live browser session.
  • Captures full-page screenshots at multiple viewport widths.
  • Extracts computed styles: colors, fonts, spacing, and radii.
  • Records hover, scroll, and click interactions.
  • Downloads public assets into public/.

Phase 2 — Foundation

  • Rewrites src/index.css with OKLCH design tokens.
  • Updates font imports and base typography.
  • Prepares the project shell for repeatable section builds.

Phase 3 — Component specs

  • Writes one spec file per section into docs/research/components/.
  • Each spec includes pixel values, states, and responsive rules.
  • Builders receive complete instructions — no inference required.

Phase 4 — Parallel build

  • Dispatches one builder per section spec.
  • Builders work in isolation to avoid conflicts.
  • Output is merged after each builder completes.

Phase 5 — Assembly and quality check

  • Assembles all built components into the main application file.
  • Runs the Vite build to confirm the project compiles without errors.
  • Checks visual fidelity against the original screenshots at multiple widths.

FAQ

Common questions

Do I need to know how to code before running the workflow?

You need to be comfortable running terminal commands and cloning a Git repository. The agent handles code generation — you do not need to write React or TypeScript manually to complete the first run.

What kind of URL should I provide?

Use the full URL of any publicly accessible website — one that opens in a browser without requiring a login. If the target page is behind authentication, the agent will not have clean access to styles and assets.

Which agent should I use?

Claude Code with --chrome is the most-tested combination. All supported agents read AGENTS.md automatically, so the workflow will run on Cursor, Windsurf, Gemini CLI, Codex CLI, Continue, Amazon Q, Augment, and OpenCode as well.

When should I start editing the generated app?

After the first run completes and you have confirmed the base clone looks correct in the browser. Verify the foundation before layering your own customizations on top.

How do I update my platform-specific rule files?

Edit AGENTS.md as your single source of truth, then run pnpm sync-rules. This regenerates the .cursor/, .windsurf/, .gemini/, and other platform folders from the shared source automatically.
